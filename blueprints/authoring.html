Authoring Blueprints

Create your own blueprints to define APIs, data models, and workflows in a declarative, language-agnostic format.

Blueprint Structure

A blueprint is a YAML file with four main sections:

SectionRequiredDescription
name / versionYesIdentity and semantic version
modelsYesData types with typed, validated fields
endpointsYesREST routes mapped to model operations
hooksNoLifecycle callbacks for custom logic

Full Example

blueprints/articles.yaml 
name: my-blueprint
version: 1.0.0
description: A custom API blueprint

# Data models with typed fields
models:
  Article:
    fields:
      id:        uuid
      title:     string, required, max:200
      body:      text, required
      author:    string, required
      published: boolean, default:false
      created:   timestamp, auto
      updated:   timestamp, auto

# REST endpoints
endpoints:
  /articles:
    GET:  list Article, paginate:true, filter:[author, published]
    POST: create Article
  /articles/:id:
    GET:    read Article
    PUT:    update Article
    DELETE: delete Article

# Hooks for custom logic
hooks:
  beforeCreate: validate, sanitize
  afterCreate:  notify

Field Types

TypeDescriptionModifiers
stringText valuerequired, min:N, max:N, pattern:regex
textLong-form textrequired
integerWhole numberrequired, min:N, max:N
floatDecimal numberrequired, min:N, max:N
booleanTrue/falsedefault:value
uuidAuto-generated unique ID—
timestampDate/timeauto (set on create/update)
enumFixed set of valuesvalues:[a,b,c]

Hooks

Hooks let server implementations inject custom logic at key points in the request lifecycle:

  • beforeCreate / afterCreate
  • beforeUpdate / afterUpdate
  • beforeDelete / afterDelete
  • beforeRead — useful for access control

Composing Blueprints

Blueprints can extend other blueprints using the extends keyword:

blueprints/blog.yaml 
name: blog
extends: api-rest
mixins:
  - auth-session
  - webhook-outbound
Test your blueprints with weblisk-server validate before deploying.
