Blueprints

Declarative specifications for APIs, data models, and workflows. Define once, implement in any language or platform.

What Is a Blueprint?

A Blueprint is a structured definition of what your server should do — not how it does it. Blueprints describe endpoints, data schemas, validation rules, relationships, and workflows in a language-agnostic format.

Think of it as an API contract that a server implementation can consume to generate boilerplate-free, spec-compliant endpoints automatically.

Declarative

Define your API surface, data models, and relationships in a structured format. No imperative code — just the contract.

Platform Independent

The same blueprint can generate a Go server, a Node.js API, a Python backend, or documentation — the implementation is separate from the definition.

Composable

Blueprints can extend and compose. Start with a base CRUD blueprint, layer on authentication, add real-time sync — each concern is a separate, reusable piece.

Versioned & Testable

Blueprints are plain files that live in your repo. Version them, diff them, review them. Every blueprint ships with a conformance test suite.

Example Blueprint

blueprints/api-rest.yaml 
name: api-rest
version: 1.0.0
description: Standard REST API with CRUD operations

models:
  Task:
    fields:
      id:        uuid
      title:     string, required, max:255
      done:      boolean, default:false
      created:   timestamp, auto

endpoints:
  /tasks:
    GET:    list Task
    POST:   create Task
  /tasks/:id:
    GET:    read Task
    PUT:    update Task
    DELETE: delete Task
