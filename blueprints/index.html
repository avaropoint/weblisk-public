Blueprints

The specification layer of the Weblisk platform. Blueprints define what the system does — architecture, protocol, patterns, agents, platforms, and project standards — without prescribing how any particular implementation achieves it.

What Is a Blueprint?

A blueprint is a structured, implementation-agnostic specification. It describes contracts, behaviors, dependencies, and verification criteria in a format that can be consumed by humans, LLMs, and automated tooling alike.

Nothing operates in the Weblisk framework without a schema-compliant blueprint. If a schema doesn't define it, the framework doesn't allow it.

Implementation-Agnostic

Blueprints describe what agents and orchestrators must do — not how. The same specs generate Go, Node.js, Cloudflare Workers, or Rust implementations.

Composable via Extends

Agents and domains inherit cross-cutting patterns (scope, policy, safety, approval, privacy) through extends — no copy-paste, no drift.

Schema-Governed

Every blueprint type has a schema with required sections, field constraints, and compliance levels — from Certified down to Rejected.

Multi-Source Resolution

The CLI resolves blueprints from local project, custom repos, and core — giving organizations full control over their specifications.

Blueprint Hierarchy

The blueprint system is organized into six categories, each governing a different layer of the platform:

Category Purpose Count
Protocol Wire protocol — endpoints, Ed25519 identity, canonical types, federation 4
Architecture System components — orchestrator, domains, agents, gateway, admin, client, observability, security 16
Patterns Cross-cutting contracts — API, auth, governance, infrastructure, workflows, operations 37
Agents Infrastructure agents — workflow, task dispatch, lifecycle, alerting, health, sync, cron 11
Platforms Implementation bindings — Go, Cloudflare Workers, Node.js, Rust 4
Standards Project conventions — structure, global config, code gen, theme, pages, components, islands 9

How Blueprints Work

Terminal 
# Generate a domain controller from blueprints
weblisk domain create seo --platform go

# Generate a work agent
weblisk agent create seo-analyzer --platform go

# Generate an application gateway
weblisk gateway create --platform go

# Validate blueprints against schemas
weblisk validate

The CLI reads blueprint specs, combines them with platform bindings, and dispatches to the configured LLM for code generation. The blueprints are the source of truth — generated code is a build artifact.

Zero-Dependency Philosophy

Blueprints enforce the Weblisk zero-dependency principle at every layer:

  • Go — Standard library only. SQLite compiled into the binary.
  • Rust — Minimal curated crates (tokio, hyper, serde). Single static binary per agent.
  • Cloudflare — Platform-native APIs only (Workers KV, Durable Objects, Web Crypto).
  • Node.js — Recommended libraries are choices, not requirements.
  • AI models — The only layer with external dependencies. Local models (Ollama) are the default.
