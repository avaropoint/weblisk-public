Agents

Agents are autonomous, specification-driven workers that execute tasks within a Weblisk application. They operate under governance policies, respect security boundaries, and can be composed into multi-agent workflows.

What Is an Agent?

An agent is a stateless or stateful worker defined in a blueprint that performs a specific task: data transformation, external API calls, scheduled jobs, event processing, or AI inference. Each agent has a declared capability set, resource budget, and governance tier.

Agents are first-class citizens in the Weblisk runtime. They are registered, monitored, and governed like any other resource — but they execute autonomously once activated.

Defining an Agent

Agents are declared in your blueprint YAML under the agents key. Each agent specifies its type, capabilities, governance tier, and resource constraints.

yaml 
agents:
  summarizer:
    type: ai
    model: gpt-4o
    capabilities: [text.summarize, text.translate]
    governance: tier-3
    budget:
      tokens_per_request: 4096
      requests_per_minute: 60
    retry: { max: 3, backoff: exponential }

The budget block enforces resource limits at runtime. If an agent exceeds its budget, the request is rejected and an alert is emitted to the governance layer.

Agent Types

Weblisk supports five built-in agent types, each optimized for a different class of workload:

Type Description
ai LLM-powered inference — text generation, summarization, classification, embeddings
transform Data mapping and ETL — reshape, validate, and enrich data between systems
webhook External HTTP callbacks — receive and process inbound webhooks from third-party services
cron Scheduled tasks — time-based execution with cron syntax for recurring jobs
event Event-driven reactive — subscribe to internal or external event streams and react in real time

Capabilities

Agents declare their capabilities explicitly. The runtime enforces these — an agent cannot access resources outside its declared scope. This principle of least privilege ensures agents remain predictable and auditable.

yaml 
agents:
  data-sync:
    type: transform
    capabilities:
      - db.read:users
      - db.write:analytics
      - http.fetch:api.example.com
    governance: tier-2

Capabilities follow a namespace.action:resource pattern. Common namespaces include db, http, text, file, and queue. Undeclared access attempts are blocked and logged.

Multi-Agent Workflows

Agents can be composed into pipelines and directed acyclic graphs (DAGs). Each step declares its inputs, outputs, and optional parallelization strategy.

yaml 
workflows:
  content-pipeline:
    steps:
      - agent: scraper
        output: raw_content
      - agent: summarizer
        input: raw_content
        output: summary
      - agent: translator
        input: summary
        output: translated
        parallel: [en, es, fr, de]

The parallel key fans out execution across multiple instances. Each variant receives the same input and produces an independent output. The runtime collects results and merges them before passing to the next step.

Governance Integration

Agents are subject to governance tiers (1–5). Higher tiers require additional checkpoints:

  • Tier 1 — No restrictions; fire-and-forget execution
  • Tier 2 — Audit logging of all inputs and outputs
  • Tier 3 — Rate limiting and behavioral monitoring
  • Tier 4 — Approval gates before execution
  • Tier 5 — Full human-in-the-loop review with signed audit trail

Governance policies are defined separately and applied to agents by tier. See the Governance documentation for full details on policy configuration.

Lifecycle

Every agent follows a defined lifecycle managed by the Weblisk runtime:

  1. Registered — Agent definition is parsed and validated from the blueprint
  2. Activated — Agent is initialized with its configuration and connected to required resources
  3. Executing — Agent is processing a task within its capability and budget constraints
  4. Completed / Failed — Task finishes successfully or encounters an unrecoverable error
  5. Deactivated — Agent is shut down and resources are released

Health checks run on a configurable interval. If an agent becomes unresponsive, the runtime applies its restart policy — either always, on-failure, or never. Failed agents emit structured error events that can trigger alerting or fallback workflows.
