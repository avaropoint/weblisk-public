Blueprints

Blueprints are declarative YAML specifications that describe your entire application — APIs, schemas, auth, integrations — in a single source of truth. Write once, generate server code for any supported language or platform.

What Is a Blueprint?

A blueprint is a YAML file that captures the full intent of a service: endpoints, data models, validation rules, middleware, auth, and deployment targets. The CLI reads it and produces production-ready code.

Instead of hand-wiring routes, controllers, and middleware across files, you describe what your service does and let Weblisk handle the how. The result is consistent, type-safe, documented code — every time.

Anatomy of a Blueprint

A blueprint file is organized into top-level sections: metadata, models, endpoints, and auth. Here's a complete example:

yaml 
blueprint:
  name: todo-api
  version: 1.0.0
  description: A simple todo API

models:
  Todo:
    fields:
      id: { type: uuid, primary: true }
      title: { type: string, required: true, max: 200 }
      completed: { type: boolean, default: false }
      created_at: { type: timestamp, auto: true }

endpoints:
  - path: /todos
    methods:
      GET: { handler: list, auth: public }
      POST: { handler: create, auth: user }
  - path: /todos/{id}
    methods:
      GET: { handler: get, auth: public }
      PUT: { handler: update, auth: owner }
      DELETE: { handler: delete, auth: owner }

auth:
  provider: jwt
  issuer: https://auth.example.com

Models

Define data shapes with built-in validation. Each model declares its fields, types, and constraints — the code generator produces structs, schemas, and database migrations from this single definition.

Fields support the following types:

  • string, number, boolean
  • uuid, timestamp
  • json, array, enum

Constraints you can apply to any field:

  • required — field must be present
  • min / max — length or value bounds
  • pattern — regex validation
  • unique — enforce uniqueness at the database level

Relationships between models use belongs_to and has_many to express ownership and collections.

Endpoints

Declare routes, HTTP methods, handlers, auth levels, and middleware. Each endpoint maps a path and method to a handler function that the generator produces.

Auth levels map to role-based guards:

  • public — no authentication required
  • user — any authenticated user
  • admin — users with the admin role
  • owner — only the resource owner

Middleware can be attached at the endpoint or method level for concerns like rate limiting, logging, or CORS.

Composition

Blueprints can import and extend other blueprints using the extends key. This enables sharing common models, auth configs, and middleware across services.

yaml 
extends:
  - ./base-auth.yaml
  - marketplace://avaropoint/rest-pagination

Local files are resolved relative to the current blueprint. Marketplace references pull shared blueprints from the Weblisk Marketplace, enabling teams to share and version common patterns.

Code Generation

Once your blueprint is defined, generate production-ready code with a single command:

bash 
weblisk generate --lang go --platform cloudflare

Built-in generation targets (extensible via custom generators):

  • Go — high-performance compiled binary
  • Rust — memory-safe systems-level output
  • Node.js — familiar JavaScript/TypeScript runtime
  • Cloudflare Workers — edge-deployed serverless

Anyone can author additional generators for other languages or platforms — the generation interface is open and documented. Use the --dry-run flag to preview generated output without writing files to disk.
