Migrations

Weblisk migrations handle schema evolution, blueprint versioning, and safe data transformations. When your models change, the CLI generates migration scripts that preserve data integrity across environments.

How Migrations Work

When you modify a blueprint's models — add or remove fields, change types, rename — running weblisk migrate diffs the current blueprint against the last known state and generates a migration file. Migrations are sequential, timestamped, and reversible.

Each migration captures both the up (apply) and down (rollback) operations. This ensures you can always return to a previous state safely, whether during development or in production hotfix scenarios.

Generating a Migration

Use the CLI to generate a new migration after modifying your blueprint models:

bash 
weblisk migrate --name add-priority-field

This generates a timestamped migration file describing the schema change:

yaml 
# migrations/20260615_001_add-priority-field.yaml
migration:
  version: 20260615_001
  description: Add priority field to Todo model
  up:
    - alter_model: Todo
      add_field:
        priority: { type: enum, values: [low, medium, high], default: medium }
  down:
    - alter_model: Todo
      remove_field: priority

Migration Types

The migration system supports a variety of schema operations:

Operation Description
add_field Add a new field with optional default value
remove_field Drop a field — data lost unless backed up
rename_field Rename a field, preserving existing data
change_type Type conversion with coercion rules
add_model Create a new model/table
remove_model Drop an entire model
add_index Create an index for query performance
seed_data Insert initial or reference data

Running Migrations

The CLI provides commands for applying, rolling back, and inspecting migration state:

bash 
# Apply all pending migrations
weblisk migrate up

# Apply only the next migration
weblisk migrate up --step 1

# Rollback the last applied migration
weblisk migrate down

# Show pending and applied migrations
weblisk migrate status

Migrations run inside transactions where the target platform supports them, ensuring atomicity — if any step fails, the entire migration is rolled back and no partial state is left behind.

Blueprint Versioning

Every blueprint change increments the version. The migration system tracks which version is deployed in each environment, preventing version skew and ensuring consistent schema state across development, staging, and production.

bash 
$ weblisk migrate status

Blueprint: todo-app v3.2.0

Environment    Version    Pending
development    3.2.0      0
staging        3.1.0      2
production     3.0.1      4

Cross-Platform

Migrations generate platform-appropriate operations depending on the deployment target. The same migration file produces different outputs depending on where it runs:

  • PostgreSQL DDL — For server deployments, migrations generate standard SQL ALTER/CREATE statements wrapped in transactions.
  • IndexedDB schema changes — For client-side stores, migrations produce version-bumped object store modifications using the IndexedDB upgrade mechanism.
  • API version negotiation — For federated services, migrations generate backward-compatible API version manifests that allow consumers to negotiate the schema version they support.

This cross-platform approach means you define schema changes once in your blueprint and the migration system handles the platform-specific details for each deployment target.
