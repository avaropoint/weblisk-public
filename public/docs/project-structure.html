Project Structure

A Weblisk project follows a convention-over-configuration layout. Whether you're building a full-stack application with blueprints or a client-only site, the directory structure keeps concerns separated and predictable.

Full-Stack Project

A project initialized with weblisk init --template full creates the following layout:

plaintext 
my-app/
├── blueprints/
│   ├── main.yaml          # Primary blueprint
│   ├── models/             # Model definitions
│   │   └── todo.yaml
│   └── agents/             # Agent definitions
│       └── summarizer.yaml
├── migrations/
│   └── 20260615_001_init.yaml
├── generated/              # CLI output (do not edit)
│   ├── go/
│   ├── rust/
│   └── workers/
├── public/                 # Client-side assets
│   ├── index.html
│   ├── css/
│   ├── js/
│   │   └── islands/
│   └── images/
├── governance/
│   └── policies.yaml
├── tests/
│   ├── blueprint.test.yaml
│   └── client/
├── weblisk.config.yaml     # Project configuration
└── package.json            # Client dependencies (optional)

Key Directories

Directory Purpose
blueprints/ YAML specs — the single source of truth for your application's models, agents, and API surface.
migrations/ Schema evolution scripts. Each migration is a timestamped YAML file that describes incremental changes to your data model.
generated/ CLI output, gitignored by default. Regenerated on every build — never edit these files directly.
public/ Static HTML, CSS, JS, and islands. Served directly by the edge runtime or your chosen platform.
governance/ Policy rules and compliance configuration. Defines rate limits, access tiers, and audit requirements.
tests/ Blueprint validation tests and client-side tests. Mirrors the source structure for discoverability.

Client-Only Project

For projects using only the client framework (no blueprints), initialized with weblisk init --template client:

plaintext 
my-site/
├── public/
│   ├── index.html
│   ├── css/
│   │   └── styles.css
│   ├── js/
│   │   ├── islands/
│   │   │   ├── counter.js
│   │   │   └── nav.js
│   │   └── state.js
│   └── images/
├── tests/
│   └── islands.test.js
├── weblisk.config.yaml
└── package.json

Configuration File

The weblisk.config.yaml file at the project root controls all aspects of code generation, client framework behavior, and governance settings:

yaml 
project:
  name: my-app
  version: 1.0.0

server:
  language: go
  platform: cloudflare
  output: ./generated

client:
  framework: true
  islands: ./public/js/islands
  styles: ./public/css

governance:
  tier: 3
  policies: ./governance/policies.yaml

The generated/ Directory

Running weblisk generate writes output to the generated/ directory. The process is deterministic — the same blueprint always produces the same code.

In most workflows this directory should be gitignored and regenerated in CI. However, you can commit it if you want to inspect diffs between generation runs or need to deploy without running the CLI.

bash 
# Regenerate all output from blueprints
weblisk generate

# Generate only for a specific language target
weblisk generate --target go

Conventions

  • Blueprints use the .yaml extension (not .yml).
  • One model per file in models/.
  • One agent per file in agents/.
  • Island files export a default enhance() function.
  • Test files mirror source structure — e.g. models/todo.yaml → tests/models/todo.test.yaml.
