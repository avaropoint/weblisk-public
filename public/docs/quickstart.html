Quick Start

From zero to production-ready in five steps. This guide walks you through installing the CLI, scaffolding a project, defining blueprints, generating code, and deploying.

1. Install the CLI

Terminal 
# macOS / Linux
$ curl -fsSL https://cdn.weblisk.dev/install.sh | sh

# or via npm
$ npm install -g @weblisk/cli

# verify
$ weblisk --version
weblisk 1.0.0

For detailed installation options (Windows, Homebrew, manual), see the Installation guide.

2. Initialize a Project

Terminal 
$ weblisk init my-app --template server
✓ Created my-app/ from weblisk-templates/server
✓ 4 blueprints, 2 agents, 1 platform target

$ cd my-app

Available templates:

Template What you get
server API server with auth, CRUD, and governance patterns
client Static site with islands, signals, and PWA
full Server + client with end-to-end blueprints
minimal Empty project with one blueprint file

This creates:

Project structure 
my-app/
├── blueprints/
│   ├── api.yaml           ← API endpoints & contracts
│   ├── auth.yaml          ← Auth patterns & policies
│   ├── governance.yaml    ← Enforcement boundaries
│   └── platform.yaml      ← Target: Cloudflare Workers
├── agents/
│   ├── domain.yaml        ← Domain agent definitions
│   └── quality.yaml       ← Quality agent definitions
├── generated/             ← Code output (after step 4)
├── weblisk.yaml           ← Project configuration
└── .env                   ← Environment variables

3. Define & Compose

Blueprints are structured YAML specifications that define your system. Anyone can read them — founders, product managers, developers, compliance officers.

blueprints/api.yaml 
kind: Pattern/API
version: 1.0.0
metadata:
  name: task-api
  description: Task management API

endpoints:
  - path: /api/v1/tasks
    methods: [GET, POST]
    auth: required
    governance:
      consent: explicit
      audit: true

  - path: /api/v1/tasks/{id}
    methods: [GET, PUT, DELETE]
    auth: required

Compose blueprints together — each one defines a separate concern (API, auth, governance, storage, webhooks). Dependencies are explicit. The framework validates composition at every stage.

Browse the Marketplace for community blueprints and templates you can fork and compose into your project.

4. Generate

Terminal 
$ weblisk generate --lang go
✓ Validated 4 blueprints (50+ checks passed)
✓ Generated 24 endpoints, 3 agents, 7 patterns
✓ Output: generated/

Generation targets:

Flag Output
--lang go Go server code with strong typing and concurrency
--lang rust Rust with zero-cost abstractions and memory safety
--lang node TypeScript/Node.js server code
--platform cf-workers Cloudflare Workers (edge deployment)

The generated code is deterministic — the same blueprint produces the same output every time. Content hashing detects post-generation tampering. You own every line.

5. Run & Deploy

Terminal 
# Start the dev server
$ weblisk dev
✓ Dev server running at http://localhost:3000
✓ 4 checkpoints active (message, storage, external, response)

# Deploy to Cloudflare Workers
$ weblisk deploy --platform cf-workers
✓ Deployed to https://my-app.weblisk.workers.dev

The dev server enforces the same governance boundaries as production — four checkpoints inspect every operation during development. No surprises at deploy time.

Client Framework Only

If you only need the client framework (islands, signals, reactive UI) without the server-side specification engine, you can use it standalone:

Terminal 
$ weblisk init my-site --template client
$ cd my-site

This scaffolds a static site with islands, styles, and a service worker. Every page is a full standalone HTML document — what you write is what ships. The framework loads via CDN import maps — no build step required.

html 
<!-- Drop an island into any HTML page -->
<div data-island="/js/islands/counter.js" data-hydrate="visible">
  <button>Count: 0</button>
</div>
js/islands/counter.js 
import { signal, effect } from 'weblisk';

export default function counter(el) {
  const [count, setCount] = signal(0);
  const btn = el.querySelector('button');

  btn.addEventListener('click', () => setCount(count() + 1));
  effect(() => { btn.textContent = `Count: ${count()}`; });
}

The island hydrates when it scrolls into view (data-hydrate="visible"). Other strategies: load (immediate), idle (when browser is idle). See Islands and Hydration for details.
