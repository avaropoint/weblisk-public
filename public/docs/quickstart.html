Quick Start

From zero to running hub in five steps. This guide walks you through installing the CLI, scaffolding a project, writing domain specs, generating your server, and running it locally.

1. Install the CLI

Terminal 
# macOS / Linux
$ curl -fsSL https://cdn.weblisk.dev/install.sh | sh

# Windows (PowerShell)
$ irm https://cdn.weblisk.dev/install.ps1 | iex

# verify
$ weblisk --version
weblisk 1.0.0

For detailed installation options (manual download, CDN-only), see the Installation guide.

2. Create a Project

Terminal 
$ weblisk new my-hub --template server/starter
✓ Created my-hub/ from weblisk-templates/server/starter
✓ .gitignore generated (secrets, keys, tokens excluded)
✓ Run: cd my-hub && weblisk doctor

$ cd my-hub

Templates are resolved from weblisk-templates. You can combine multiple:

Terminal 
# Combine a client + server template (files are merged)
$ weblisk new my-app --template client/blog --template server/starter

Available templates:

Template What you get
client/starter Client-only site with islands, signals, PWA, and service worker
server/starter Extends client/starter with a hub, orchestrator, gateway, domains, and agents

This creates:

Project structure 
my-hub/
├── .weblisk/
│   ├── config.yaml          ← Hub configuration
│   └── secrets/             ← Local secrets (0600 permissions)
├── domains/
│   └── seo/
│       └── domain.yaml      ← Domain controller spec
├── agents/
│   ├── seo-analyzer/
│   │   └── agent.yaml       ← Work agent spec
│   └── cron/
│       └── agent.yaml       ← Infrastructure agent spec
├── public/                  ← Static client files (optional)
├── .gitignore               ← Auto-excludes secrets, keys, tokens
└── README.md

3. Define Domains & Agents

Blueprints are YAML specs that describe your system. The CLI reads these and dispatches to the configured LLM for code generation — no boilerplate, no copy-paste.

domains/seo/domain.yaml 
name: seo
version: 1.0.0
description: SEO analysis and optimization domain

required_agents:
  - seo-analyzer
  - a11y-checker

workflows:
  - name: seo-audit
    trigger: action=audit
    phases:
      - name: scan
        agent: seo-analyzer
      - name: accessibility
        agent: a11y-checker
      - name: report
        agent: seo-analyzer

Each agent has its own spec declaring capabilities, secrets, and task handling:

agents/seo-analyzer/agent.yaml 
name: seo-analyzer
type: work
domain: seo
version: 1.0.0

capabilities:
  - file:read **/*.html
  - llm:chat *

secrets:
  - LLM_API_KEY
Browse the Marketplace for pre-built blueprints and capabilities you can purchase or fork.

4. Generate & Validate

Terminal 
# Validate project health
$ weblisk doctor
✓ .gitignore excludes secrets
✓ .weblisk/config.yaml valid
✓ Blueprint YAML valid (2 domains, 3 agents)
⚠ Secret 'LLM_API_KEY' declared but not set

# Set required secrets
$ weblisk secret set seo-analyzer LLM_API_KEY
Enter value: ********
✓ Secret stored: .weblisk/secrets/seo-analyzer/LLM_API_KEY

# Generate server implementation from specs
$ weblisk server init --platform go
Reading .weblisk/config.yaml, domains/*, agents/*...
Dispatching to LLM for code generation...
✓ Generated orchestrator, gateway, 1 domain, 2 agents
✓ Platform: Go

Platform targets:

Flag Output
--platform go Go server — strong typing, concurrency, small binaries
--platform node Node.js — TypeScript, fast iteration
--platform rust Rust — zero-cost abstractions, memory safety
--platform cloudflare Cloudflare Workers — edge deployment, global distribution

You can also generate individual agents or domains:

Terminal 
$ weblisk agent create seo-analyzer --platform go
$ weblisk domain create seo --platform go
$ weblisk gateway create --platform go

5. Run

Terminal 
# Start the full hub (builds + runs all components)
$ weblisk server start
✓ Orchestrator running on :9800
✓ Gateway running on :9900
✓ Domain 'seo' running on :9700
✓ Agent 'seo-analyzer' registered
✓ Agent 'cron' registered

# Or start the dev server (file watching + live reload)
$ weblisk dev --port 3000

# Verify everything is healthy
$ weblisk server verify
✓ Orchestrator healthy (http://localhost:9800)
✓ 2 agents registered, 2 online

6. Set Up Operator Identity (Optional)

To manage a running hub via CLI operations commands, generate your ML-DSA-65 operator identity:

Terminal 
# Generate operator key pair (encrypted with passphrase)
$ weblisk operator init
Enter passphrase (min 12 characters): ********
✓ Generated key pair in ~/.weblisk/keys/

# Register with your hub
$ weblisk operator register --orch http://localhost:9800
✓ Registered as admin (first operator — auto-approved)
  Token stored: ~/.weblisk/token

# Now you can use operations commands
$ weblisk status
$ weblisk agents list
$ weblisk approvals list

Client Framework Only

If you only need the client framework (islands, signals, reactive UI) without the hub server, scaffold a client-only project:

Terminal 
$ weblisk new my-site --template client/default
$ cd my-site
$ weblisk dev

This scaffolds a static site with islands, styles, and a service worker. Every page is a full standalone HTML document — what you write is what ships. The framework loads via CDN import maps — no build step required.

html 
<!-- Drop an island into any HTML page -->
<div data-island="/js/islands/counter.js" data-hydrate="visible">
  <button>Count: 0</button>
</div>
js/islands/counter.js 
import { signal, effect } from 'weblisk';

export default function counter(el) {
  const [count, setCount] = signal(0);
  const btn = el.querySelector('button');

  btn.addEventListener('click', () => setCount(count() + 1));
  effect(() => { btn.textContent = `Count: ${count()}`; });
}

The island hydrates when it scrolls into view (data-hydrate="visible"). Other strategies: load (immediate), idle (when browser is idle). See Islands and Hydration for details.
