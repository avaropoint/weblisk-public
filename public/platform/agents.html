Agents

Every agent is an autonomous HTTP service that registers with the orchestrator, declares its capabilities, and collaborates through events and messages. The framework includes 11 infrastructure agents — you build the work agents.

The Agent Model

Every agent — infrastructure or work — follows the same model:

  • Serves exactly 6 HTTP endpoints: describe, execute, message, event, health, shutdown
  • Registers capabilities and namespace ownership with the orchestrator
  • Publishes and subscribes to namespaced, scoped events
  • Holds its own storage (declared tables only)
  • Operates independently — no shared memory, no direct function calls between agents

Infrastructure Agents

11 infrastructure agents provide hub-wide services. They are not domain-specific — they serve all domains and work agents equally.

Work Agents

Work agents contain your business logic. They are defined by domain controllers and serve domain-specific tasks. Examples:

  • SEO analyzer agent — crawls pages, scores performance, suggests improvements
  • Content manager agent — manages blog posts, pages, media
  • Auth handler agent — user registration, login, session management
  • Payment agent — billing, subscriptions, invoice generation
  • Chat agent — real-time messaging, presence, history

Each work agent follows the same 6-endpoint model and registers with the orchestrator exactly like an infrastructure agent.

Agent Registration Flow

  1. Agent starts and generates (or loads) its ML-DSA-65 key pair
  2. Agent sends POST /v1/register to the orchestrator with its manifest and public key
  3. Orchestrator verifies identity, validates manifest, checks namespace availability
  4. Orchestrator assigns a WLT token and adds agent to the service directory
  5. Orchestrator publishes system.agent.registered event
  6. Agent begins serving requests on its 6 endpoints
