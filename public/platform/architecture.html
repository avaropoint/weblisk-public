Architecture

The Orchestrator

The orchestrator is the trust anchor and registry service for every hub. It manages agent lifecycle without executing agent logic. Its responsibilities:

  • Agent registration — verifies ML-DSA-65 identity, validates manifests, assigns tokens
  • Service directory — maintains routing table and namespace map
  • Namespace control — enforces exclusive namespace ownership at registration
  • Channel brokering — establishes direct agent-to-agent communication channels
  • Auth enforcement — verifies WLT tokens on every authenticated request
  • Audit logging — append-only log of all registration, deregistration, and security events
  • Health aggregation — monitors agent health across the hub
  • System events — publishes system.* events for agent lifecycle

Startup Sequence

  1. Load or generate ML-DSA-65 identity key pair
  2. Initialize empty registry
  3. Reserve system namespace (built-in events)
  4. Register HTTP routes (6 orchestrator endpoints)
  5. Start server

The orchestrator does NOT execute agent logic, route tasks, manage strategies, store observations, or handle approvals. Those concerns belong to infrastructure agents (Workflow, Task, Lifecycle).

Two Gateways, Complete Separation

Every hub has two gateways that serve fundamentally different audiences with completely separate security boundaries:

App Gateway Admin Gateway
Serves End users Operators
Auth User sessions, OAuth, MFA Operator identity, key-based
Route protection ABAC, rate limiting Operator roles, audit logging
Compromise impact User-facing only Unreachable from app gateway

The two gateways share no session state, no authentication flow, no network listener, and no cookie domain. If the app gateway is compromised, the admin gateway is unreachable.

Six Trust Boundaries

The threat model identifies 6 trust boundaries — transitions where security context changes and enforcement must validate:

  1. Browser ↔ App Gateway — untrusted external clients cross into the hub
  2. App Gateway ↔ Agents — requests enter the internal agent network
  3. Agents ↔ Storage — agents access the persistence layer
  4. Operator ↔ Admin Gateway — privileged access to hub administration
  5. Hub ↔ Hub Federation — cross-organization agent invocation
  6. Enforcement Layer — the policy decision boundary itself

Every boundary has defined threats, mitigations, and monitoring requirements. See Security & Governance for the full enforcement architecture.
