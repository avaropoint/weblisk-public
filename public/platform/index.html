The Weblisk Platform

A complete specification corpus that defines how enterprise software is built — from wire protocol to marketplace. Every layer is open, documented, and interconnected.

The Weblisk platform is not a library you install. It is a specification framework — a structured corpus of interlocking blueprints that define your system's architecture, security model, data contracts, and deployment targets. Domain-expert agents consume those blueprints and generate production code for any supported language or platform that you fully own. Built-in generators ship for Go, Rust, Node.js, and Cloudflare Workers — and the generation interface is open for anyone to extend.

This section explores every layer of the framework in depth. The documentation shows you how to build. The platform pages show you what you're building on and why each layer exists.

Hub Architecture

Every Weblisk deployment is a hub — a self-sovereign node that runs your orchestrator, agents, and services. The hub is the unit of deployment, the unit of trust, and the unit of federation.

Hub Architecture 
Hub
├── App Gateway          ← Security boundary for end users
├── Admin Gateway        ← Separate security boundary for operators
├── Orchestrator         ← Trust anchor, agent registry, namespace control
│   ├── Domain Controllers
│   │   ├── Work Agents  ← Your business logic (SEO, content, auth...)
│   │   └── Infra Agents ← 11 built-in (workflow, task, lifecycle...)
│   └── Enforcement      ← Non-bypassable policy boundary
├── Marketplace          ← Blueprint catalog and distribution
├── Federation           ← Hub-to-hub collaboration protocol
└── Data Security        ← Encryption, classification, sovereignty

The orchestrator is the trust anchor — it manages agent registration, identity verification, namespace ownership, service directory distribution, and channel brokering. It does not execute agent logic, route tasks, or store business state. Those concerns belong to infrastructure agents that register like any other agent.

Seven Specification Categories

The framework is organized into seven specification categories. Each defines a different aspect of the system. Together they form a complete, interlocking corpus.

Category Specs Defines
Protocol 4 HTTP+JSON wire format, agent endpoints, identity & signing, type system, federation wire protocol
Architecture 16 Orchestrator, domain controllers, gateways, hub, storage, CLI, admin, observability, lifecycle, testing, change management, enforcement, data security, threat model
Agents 11 Infrastructure agents: workflow, task, lifecycle, alerting, incident-response, health-monitor, hub, sync, cron, webhook, email-send
Patterns 37 Cross-cutting contracts across 6 categories: API & Communication, Auth & Security, Governance, Infrastructure, Workflows, Operations
Schemas 10 Blueprint validation, compliance levels (Certified → Rejected), content hashing, version constraints
Standards 9 Project structure, global identity, code conventions, theme tokens, pages, components, islands, assets, connections
Platforms 4 Generation targets: Go, Cloudflare Workers, Node.js, Rust — standard library first, explicit dependencies only when blueprint requires

How Blueprint-Driven Development Works

There is no traditional build step. The LLM is the compiler. YAML is the source code. HTML, CSS, and JS are the compiled output.

Generation Pipeline 
Developer writes YAML → LLM generates code → Output serves directly
         ↑                                            |
         └────── Change blueprint, regenerate ────────┘

Pipeline inputs (in order):
1. global.yaml   — Project identity, brand, policies (loaded first, always)
2. code.yaml     — Code conventions (loaded second, constrains all output)
3. theme.yaml    — Design tokens derived from global brand
4. blueprint     — The target blueprint (page, component, island, agent...)
5. patterns      — Cross-cutting contracts that constrain generation

Three Project Modes

Mode What You Describe What You Get
Client-only global + code + pages Static HTML/CSS/JS, CDN import map
Client + Islands Above + islands Static site + interactive regions bound to agents
Full stack Above + domains + agents Complete hub with client and server

A portfolio is 3 blueprints. A SaaS app is 30. The structure scales because each blueprint is self-contained.

Key Design Principles

Zero Dependencies

Every platform target prioritises native and standard library APIs first. External frameworks or libraries are only introduced when a blueprint explicitly requires functionality that cannot be achieved with the standard library alone — and every such dependency must be declared in the blueprint. No implicit package manager sprawl, no undeclared bindings, no external database servers.

Post-Quantum Cryptography

Weblisk adopts NIST's post-quantum cryptography standards (released August 2024). All classical algorithms (RSA, ECDSA, Ed25519) are excluded entirely.

Purpose Algorithm Standard
Digital signatures ML-DSA-65 (CRYSTALS-Dilithium) FIPS 204
Key encapsulation ML-KEM-768 (CRYSTALS-Kyber) FIPS 203
Backup signatures SLH-DSA-SHA2-128s (SPHINCS+) FIPS 205
Symmetric encryption AES-256-GCM FIPS 197
Key derivation Argon2id RFC 9106

Non-Bypassable Enforcement

The enforcement layer sits between agents and every resource they consume. An agent cannot reach the message bus, storage layer, or external services without passing through enforcement. Four boundaries — message, storage, external, response — inspect every operation regardless of agent cooperation. Learn more →

Federation by Consent

Hubs collaborate through the federation protocol with explicit, non-transitive, time-limited trust relationships. Data contracts declare the minimum viable data set for each collaboration — fields not in the contract are stripped before crossing the boundary. Data sovereignty is a structural constraint, not a policy afterthought. Learn more →
