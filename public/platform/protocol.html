Protocol

The universal wire protocol for all Weblisk agents and orchestrators. Every implementation — regardless of language or platform — conforms to this specification exactly. All communication is HTTP + JSON.

Design Principles

  1. One protocol — HTTP is the only communication layer. No separate message broker, queue, or bus infrastructure.
  2. Event-driven — Agents coordinate through namespaced, scoped events delivered via HTTP POST. Workflows, tasks, and lifecycle loops are all event-driven.
  3. Namespace ownership — Every event namespace is exclusively owned by one agent. The orchestrator enforces this at registration, preventing collisions.
  4. Scoped delivery — Events carry a scope field that limits delivery to the intended recipient. Agents only receive events addressed to them unless they have explicit permission to observe globally.
  5. Separation of concerns — The orchestrator manages agent lifecycle, discovery, trust, and namespace control. It does NOT execute tasks, manage workflows, or store business state.

Protocol Specifications

The protocol layer consists of four specifications:

Specification Defines
spec The agent protocol — registration, execution, messaging, events, health, channels
identity Cryptographic identity — ML-DSA-65 key pairs, signing, verification, Weblisk Tokens (WLT)
types Shared type definitions — AgentManifest, TaskRequest, TaskResult, EventEnvelope, ErrorResponse
federation Multi-orchestrator collaboration — cross-tenant agent invocation, data contracts, trust tiers

Agent Endpoints

Every agent serves exactly 6 endpoints on a single HTTP port. All endpoints use POST for uniformity — simplifying agent implementation to a single HTTP handler dispatch.

Endpoint Auth Purpose
POST /v1/describe No Returns agent identity, capabilities, event subscriptions, namespace ownership
POST /v1/execute Yes Executes a task synchronously — the core work endpoint
POST /v1/message Yes Receives direct agent-to-agent messages
POST /v1/event Yes Receives scoped events from subscribed namespaces
POST /v1/health No Returns agent health status and metrics
POST /v1/shutdown Yes Graceful shutdown signal from orchestrator

Orchestrator Endpoints

The orchestrator exposes exactly 6 endpoints:

Endpoint Auth Purpose
POST /v1/register Identity* Agent registration with ML-DSA-65 identity verification
DELETE /v1/register Yes Agent deregistration
GET /v1/services Yes Service directory — routing table and namespace map
POST /v1/channel Yes Request direct agent-to-agent channel
GET /v1/health No Orchestrator and hub health
GET /v1/audit Yes Query append-only audit log

* Registration uses ML-DSA-65 identity verification instead of tokens.

Conventions

  • All paths are prefixed with /v1
  • All request/response bodies are application/json
  • All timestamps are Unix epoch seconds (int64)
  • All signatures are base64url-encoded ML-DSA-65 signatures (3309 bytes decoded)
  • All public keys are base64url-encoded ML-DSA-65 public keys (1952 bytes decoded)
  • IDs are 32-character hex strings (16 random bytes); event IDs use UUID v7
  • Unknown JSON fields MUST be ignored (forward compatibility)
  • Request bodies limited: 1 MB default, 10 MB for task execution

Identity & Cryptography

Every entity — agent or orchestrator — generates an ML-DSA-65 key pair on first run. This key pair is the foundation of all identity, signing, and token operations.

Post-Quantum Algorithms

Purpose Algorithm Standard
Digital signatures ML-DSA-65 (CRYSTALS-Dilithium) FIPS 204
Key encapsulation ML-KEM-768 (CRYSTALS-Kyber) FIPS 203
Backup signatures SLH-DSA-SHA2-128s (SPHINCS+) FIPS 205
Symmetric encryption AES-256-GCM FIPS 197
Key derivation Argon2id RFC 9106

Weblisk does not use any quantum-vulnerable algorithms. No RSA, no ECDSA, no Ed25519, no classical Diffie-Hellman. Per NIST IR 8547, all elliptic curve and RSA algorithms will be deprecated by 2030 and removed by 2035.

Weblisk Tokens (WLT)

Token format: base64url(header).base64url(payload).base64url(signature)

  • All three parts use base64url encoding WITHOUT padding (RFC 4648 §5)
  • Algorithm header: alg field MUST be ML-DSA-65; any other value is rejected
  • Claims: sub (agent name), iss ("orchestrator"), cap (capabilities)
  • Expiry: 24 hours (configurable)
  • JSON signing uses RFC 8785 JSON Canonicalization Scheme (JCS)

Key Management

  • Generate fresh ML-DSA-65 key pair on first run using cryptographically secure random source
  • Key sizes: 1952-byte public key, 4032-byte private key, 3309-byte signature
  • Private keys stored encrypted at rest using AES-256-GCM with Argon2id-derived key
  • On startup: load existing keys if present, generate new ones if absent

Federation Protocol

Federation enables businesses to expose specific agent capabilities to partners, customers, and the wider hub network — without exposing internal systems, data, or infrastructure.

Design Principles

  1. Zero trust by default — No orchestrator trusts another until an explicit trust relationship is established and cryptographically verified. Trust is never implied by network proximity.
  2. Data minimization by construction — Data contracts declare the minimum viable data set for each collaboration. Fields not in the contract are stripped before crossing the boundary. Enforced at the protocol level.
  3. Behavioral integrity — Agents are fingerprinted at registration. Changes to manifest, capabilities, or behavioral patterns are detected and flagged before they can affect cross-boundary interactions.
  4. Sovereignty-aware routing — Data residency requirements are first-class. Tasks involving regulated data are only routed to agents within the permitted jurisdiction.
  5. Non-repudiation — Every cross-boundary message is signed by both sending and receiving orchestrator. Full chain of custody is auditable.

Federation Conventions

  • All federation endpoints prefixed with /v1/federation
  • Trust relationships are explicit, non-transitive, and time-limited
  • Data contracts are fail-closed — unknown fields are rejected, not ignored
  • Cross-boundary messages signed by both orchestrators
  • Federation topic qualification uses :: separator (e.g., acme-corp::workflow.completed)
