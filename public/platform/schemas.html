Schemas

Every blueprint receives a compliance level after automated validation. The level determines what the blueprint is allowed to do in the framework. Non-compliant blueprints are flagged, restricted, or rejected depending on severity.

Foundational Rule

Nothing operates in the Weblisk framework without a schema-compliant blueprint. If a schema doesn't define it, the framework doesn't allow it.

This applies to agent capabilities, event namespaces, storage tables, API endpoints, workflow definitions, configuration parameters, and inter-agent communication patterns.

Compliance Levels

Level Badge Description Key Permissions
Certified ✅ Fully compliant. All sections, fields, security declarations, dependency contracts. Register, publish to marketplace, consumed by others, full messaging, events, storage, scheduling
Compliant ⚠️ Structurally valid. Minor issues (missing optional sections, incomplete tests). Register, full messaging, events, storage, scheduling. Cannot publish to marketplace.
Restricted 🟡 Partially valid. Validation errors, missing fields, incomplete security. Register, own storage only. Limited messaging (declared collaborators only), limited events (own namespace, system.* only).
Non-Compliant 🔴 Failed critical validation. Missing required sections, undeclared capabilities. Sandbox mode only. Own storage (sandboxed). Fully isolated from other agents.
Rejected ❌ Cannot operate. Structural integrity failure, signature mismatch, security violations. Cannot register. Cannot operate.

Schema Files

10 schemas govern every blueprint type in the framework:

Schema Governs Applies To
common Shared rules inherited by all blueprint types Every blueprint
compliance Compliance levels, validation, enforcement, security boundaries Every blueprint
config Hub configuration file (.weblisk/config.yaml) Every project
agent Infrastructure agent blueprints agents/ directory
domain Domain controller blueprints Domain YAML specs
protocol Wire protocol specifications protocol/ directory
pattern Cross-cutting patterns patterns/ directory
architecture System architecture components architecture/ directory
platform Platform implementation bindings platforms/ directory
standard Project-level blueprint standards standards/ directory

Blueprint Types

Six blueprint types are governed by the schema system:

  • agent — Infrastructure agent specifications (22 required sections)
  • domain — Domain controller specifications
  • protocol — Wire protocol specifications
  • pattern — Cross-cutting implementation contracts
  • architecture — System architecture components
  • platform — Platform implementation bindings (built-in: Go, Cloudflare, Node.js, Rust — extensible)

Blueprint Resolution Order

When the framework resolves a blueprint reference, it searches in priority order:

  1. Local project — blueprints/ in the current project
  2. Custom sources — registered external blueprint sources
  3. Core — the weblisk-blueprints repository (framework defaults)

Local always wins. This allows projects to override any framework blueprint while maintaining compatibility.

Validation Pipeline

Every blueprint goes through multi-phase validation before receiving a compliance level:

  1. Structural validation — all required sections present, frontmatter valid
  2. Field validation — all fields match expected types and constraints
  3. Security validation — access control, capabilities, scope declarations complete
  4. Dependency resolution — all referenced patterns, schemas, and collaborators exist
  5. Content hashing — generates integrity hash for the validated blueprint
  6. Compliance assignment — assigns one of 5 compliance levels based on results
