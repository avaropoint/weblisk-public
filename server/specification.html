Server Specification

The formal contract that every Weblisk server implementation must fulfil.

Overview

The Weblisk Server Specification defines a minimal set of endpoints, message formats, and lifecycle behaviours. Any server that implements this spec is compatible with the Weblisk client framework, any blueprint, and any agent.

Design Principles

  • Minimal surface — only endpoints that every implementation needs
  • Transport agnostic — works over HTTP/1.1, HTTP/2, or HTTP/3
  • Discovery-based — clients discover capabilities via a well-known manifest
  • Versioned — every release is semantically versioned with backward compatibility guarantees

Required Endpoints

EndpointMethodDescription
/.well-known/weblisk/healthGETHealth check — returns status and version
/.well-known/weblisk/manifestGETServer capabilities, loaded blueprints, active agents
/api/syncPOSTClient-to-server data synchronisation
/api/rpcPOSTBlueprint-generated RPC dispatch

Manifest Format

specification.yaml 
# Weblisk Server Specification v1.0

version: 1.0.0
status: stable

endpoints:
  health:
    method: GET
    path:   /.well-known/weblisk/health
    response: { "status": "ok", "version": "1.0.0" }

  manifest:
    method: GET
    path:   /.well-known/weblisk/manifest
    description: Returns server capabilities and loaded blueprints

  sync:
    method: POST
    path:   /api/sync
    description: Client-to-server data synchronisation

Lifecycle

A compliant server follows this lifecycle:

  1. Boot — load configuration and validate environment
  2. Load Blueprints — parse and register all API blueprints
  3. Start Agents — initialise and connect all registered agents
  4. Listen — begin accepting requests
  5. Shutdown — gracefully drain connections and stop agents
The specification is versioned alongside the framework. Breaking changes only occur in major versions.
